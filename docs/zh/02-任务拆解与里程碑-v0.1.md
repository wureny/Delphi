# Polymarket Ontology 任务拆解与里程碑（v0.2）

## 1. Epic 结构
### Epic A: Ontology Domain Modeling
目标：定义实体、关系、字段与约束。

建议 Issue：
1. A1 定义核心实体字典（Event/Market/Outcome/...）
2. A2 定义关系图与 cardinality
3. A3 定义字段标准（命名、类型、时间、货币）
4. A4 定义校验规则与错误码
5. A5 定义市场微观结构实体（OrderBookSnapshot/TradePrint/MarketMicrostructureState）

### Epic B: Data Mapping Pipeline
目标：把 Polymarket 原始结构映射到 ontology schema。

建议 Issue：
1. B1 采样并整理输入数据样本集
2. B2 编写 mapping spec（字段映射表）
3. B3 实现映射器 v0
4. B4 增加异常数据处理路径
5. B5 接入 CLOB / WebSocket 的订单簿与成交数据
6. B6 实现 market metadata + microstructure 的多源合并

### Epic C: Validation & QA
目标：确保结果可信、可追溯、可回归。

建议 Issue：
1. C1 实现 schema 校验器
2. C2 构建测试样例库（正常/异常）
3. C3 输出校验报告与统计指标
4. C4 建立回归测试流程
5. C5 增加薄盘口/宽 spread/低成交确认的专门校验样例

### Epic D: Documentation & Governance
目标：让协作像团队项目一样可持续。

建议 Issue：
1. D1 完善 PRD 与技术文档
2. D2 建立 issue 模板和标签规范
3. D3 建立 milestone 看板规则
4. D4 建立架构决策记录（ADR）模板

### Epic E: Evaluation Benchmark（核心）
目标：验证 ontology 是否显著提升 agent 对市场理解能力。

建议 Issue：
1. E1 定义评测任务集（检索/理解/关系追踪）
2. E2 构建 Raw Dataset Agent 基线流程
3. E3 构建 Ontology Agent 对照流程
4. E4 输出对照评测报告与结论
5. E5 增加“薄盘口鲁棒性”评测：Agent 是否会被异常价格跳动误导

### Epic F: Visualization & Editable Ontology
目标：实现 ontology 关系可视化与受控编辑能力。

建议 Issue：
1. F1 定义 JSON -> Graph 映射规范
2. F2 实现 Neo4j 导入导出脚本
3. F3 实现基础关系可视化页面/脚本
4. F4 定义编辑操作约束（防止破坏 schema）

### Epic G: Trading & Execution Ontology
目标：让 agent 的结论可以进入风控可审计的交易执行语义链路。

建议 Issue：
1. G1 定义 Portfolio/Order/Execution/RiskPolicy 等实体
2. G2 构建交易域 JSON Schema
3. G3 构建 decision -> order -> execution 样本链路
4. G4 定义并实现风险门禁规则（含人工审批）

## 2. 建议里程碑
1. Milestone 1（第 1-2 周）：完成 Epic A + B1/B2 + A5
2. Milestone 2（第 3-4 周）：完成 Epic B3/B4/B5/B6 + C1/C2
3. Milestone 3（第 5-6 周）：完成 C5 + E 系列与 F1/F2
4. Milestone 4（第 7 周）：完成 F3/F4 + D 系列收口
5. Milestone 5（第 8-9 周）：完成 G 系列并打通 paper trading 闭环

## 4. Definition of Done（DoD）
1. 代码或文档已提交并通过自检。
2. 验收标准在 issue 中被逐条勾选。
3. 相关文档（PRD/映射规范/进展）已更新。
4. issue 关联到正确 milestone 与 label。
5. 若是评估 issue，必须包含与基线对照结果。
