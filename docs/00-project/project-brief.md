# Project Brief

这是项目总入口。先不要追求完整，先把方向说清楚。

## 1. Project Snapshot

- 项目名: Delphi
- 一句话描述: 多agent系统，基于ontology，利用到context graph等技术，当前v0版本计划成为散户分析美股标的的最佳选择
- 项目类型: 毕设 / AI product
- 当前阶段: idea / redesign / prototype

## 2. Why This Project

- 这个项目要解决什么真实问题
当前针对“某只股票现在值不值得买”这类问题，市场上的分析方式大多存在两个极端：
一种是传统投研流程，依赖人工在价格、新闻、财报、宏观政策、市场情绪等多个信息源之间手动切换、整理和综合，效率低，且难以持续更新；
另一种是直接用单一大模型或搜索工具生成一份分析报告，虽然快，但通常缺乏结构化的对象表示、缺乏清晰的推理路径，也难以维护“为什么得出这个结论”的上下文。
本项目要解决的核心问题是：如何将异构金融信息组织成统一的 investment case，并通过多 agent 协作，生成更可解释、可追踪、可持续更新的投资判断。

- 为什么这个问题值得做成一个 AI 产品，而不是普通软件
因为这个问题的核心不只是“展示数据”，而是“理解问题并组织判断”。
普通软件适合做行情展示、指标计算、资讯聚合，但难以完成以下任务：
	1.	将用户自然语言问题转化为结构化的研究任务；
	2.	在不同信息源之间动态提取与当前 case 最相关的 context；
	3.	让不同 agent 分别承担 thesis construction、risk analysis、liquidity interpretation、signal fusion 等认知任务；
	4.	根据上下文动态选择合适的 skills，并输出带有 reasoning trace 的研究结果。
因此，这更适合做成一个 AI-native 的研究系统，而不是一个静态信息平台。

- 为什么现在做
现在做这个项目有三个原因。
第一，大模型和 agent framework 已经使“多步骤研究任务自动化”成为可能，不再只是理论讨论。
第二，金融研究场景中的信息复杂度越来越高，仅靠人工或单次问答式 AI 输出，已经难以支撑高质量判断。
第三，关于 ontology、knowledge graph、context graph、agent memory 与 skill routing 的讨论正在快速升温，但多数讨论停留在概念层，缺少一个小而具体、能够运行的原型系统来展示它们如何协同工作。
因此，现在是一个适合用 v0 原型去验证这套方法论的时点。


## 3. Target Users

- 核心用户是谁
核心用户是需要快速形成结构化投资判断的二级市场研究者与交易者，尤其包括：
	1.	有一定金融基础、但缺乏系统化研究框架的个人投资者；
	2.	需要高频处理信息、希望用 AI 辅助 research workflow 的交易员或研究员；
	3.	对 agentic research、AI-native investing、context engineering 感兴趣的 builder / researcher。
在毕设 v0 阶段，最核心的目标用户可以先聚焦为：
希望快速获得“结构化股票研究结论”的个人研究者和 AI-native 投资实验者。

- 用户当前是怎么解决这个问题的，为什么还不够好
当前用户通常有几种做法：
	1.	自己手动查价格、新闻、财报、宏观数据，再凭经验做判断；
	2.	使用雪球、TradingView、券商软件、财经媒体等工具分别看不同信息；
	3.	直接问 ChatGPT / Claude 一次性生成股票分析报告。
这些方式的问题在于：
	•	信息来源分散，用户需要自己完成整合；
	•	很难把“价格、新闻、宏观、流动性、市场预期”放进同一个统一框架里；
	•	单次 AI 报告往往是一次性的，难以持续更新；
	•	缺少清晰的对象层和决策路径，用户无法知道结论到底建立在哪些证据和假设上；
	•	当市场变化时，系统无法基于已有 investment case 进行状态更新，只能重新来一遍。


## 4. Core Value Proposition

- 用户使用前
用户面对的是碎片化信息流：价格在一个地方，新闻在一个地方，宏观在一个地方，市场情绪和预期在另一个地方。
即使使用大模型，也往往只能得到一份“看起来合理”的分析文字，但很难知道：
	•	结论基于哪些核心 thesis；
	•	哪些证据支持，哪些证据反驳；
	•	当前的流动性环境是否支持重估；
	•	哪些风险会让判断失效；
	•	如果未来有新事件发生，该如何更新当前结论。

- 用户使用后
用户可以直接提出一个自然语言问题，例如“这只股票未来 3 个月值不值得买”，系统会自动围绕该资产创建一个结构化的 investment case，并基于 ontology 统一组织：
	•	thesis
	•	evidence
	•	risk
	•	liquidity / macro context
	•	microstructure signals
	•	final judgment
同时，系统使用 context graph 记录 agent 的任务拆解、skill 调用、context 选择和中间判断路径，使结果不只是“一个答案”，而是一套可以追踪、更新和复用的研究对象。

- 这个产品最不可替代的一点
它将投资判断从一次性的文本生成，变成了一个可结构化表示、可追踪演化、可由多 agent 协同维护的 investment case；
也就是说，它并非只是在简单地回答问题，而是在维护一个动态的 research object。

## 5. Scope

### In Scope

	•	一个面向股票研究问答的 AI 产品界面，支持用户通过自然语言提问，例如“这只股票未来 3 个月值不值得买”
	•	左侧以对话为核心的主交互区，负责承接用户提问、系统回复与研究结果展示
	•	右侧可展开的 agent canvas，用多个mac 终端的可视化方式展示多个 agent 的运行过程
	•	将系统内部的关键过程以可理解、可观察的方式呈现出来，包括：
	•	ontology object ingestion / mapping
	•	skill usage
	•	task decomposition
	•	intermediate findings
	•	decision trace / context graph
	•	用多个 agent terminal 的形式表现多 agent 协作，每个 terminal 对应一个 agent 的工作视角与状态
	•	支持用户在“只看答案”和“观察系统如何工作”之间切换
	•	输出结构化研究结果，而不仅仅是一段自由文本
	•	整体产品风格采用暗黑、极简、科技感导向，强调流畅与可展示性
	•	产品形态同时服务于两类目标：
	•	实际使用中的 research assistant 体验
	•	毕设答辩中的系统可视化展示效果

### Out of Scope

	•	面向专业交易团队的完整交易终端
	•	自动下单、组合管理、风控执行等交易基础设施
	•	高密度 Bloomberg 式信息终端界面
	•	复杂 BI 仪表盘、重图表化、多窗口金融工作站体验
	•	面向所有资产类别的大而全研究平台
	•	过度暴露底层技术细节，导致产品界面变成工程调试面板
	•	将可视化部分做成炫技式系统监控，而不是服务于用户理解的产品体验
	•	在 v0 阶段支持完全自由配置 agent 数量、工作流与策略系统


## 6. Product Form
•	产品整体形态:
产品整体采用“双栏结构”：左侧是对话与结果，右侧是可折叠的 agent canvas。
左侧负责承接用户的问题与最终结果，保持类似 ChatGPT 的极简交互习惯；
右侧负责把系统内部的多 agent 协作过程具象化，让用户可以看到这不是一次普通的文本生成，而是一个由多个研究 agent 协同完成的分析流程。
•	为什么采用这种形态:
因为这个项目的独特价值，不只是“回答一个股票问题”，而是“展示一个基于 ontology 和 context graph 的多 agent 系统如何工作”。
如果只有聊天界面，用户很难感受到系统内部的结构化与协作；
如果只有可视化界面，又会损失 AI 产品最自然的交互入口。
因此，双栏结构是同时满足产品可用性和系统可展示性的最佳折中。


## 7. Core User Experience
•	默认体验:
用户进入产品后，首先看到的是一个极简问答界面，可以直接输入问题，例如：
“AAPL 未来三个月值不值得买？”
系统开始分析，并在左侧生成逐步展开的回答，在右侧同步展示多 agent 的工作过程。
•	右侧 canvas 的角色:
右侧不是单纯的动画区域，而是系统的“可观察工作台”。
它以 4 个风格统一的 mac风格 terminal形式展示 4 个 agent，每个 agent 都有自己的状态、任务、skill 调用和中间输出。
用户可以看到：
•	哪个 agent 正在工作
•	当前在处理什么子任务
•	用到了哪些 skills
•	哪些 ontology objects 被提取或更新
•	当前 decision trace 走到了哪一步
•	收起与展开:
对普通使用者来说，右侧 canvas 可以收起，产品退化为纯问答模式；
对进阶用户或答辩展示场景来说，右侧 canvas 可以展开，完整展示多 agent 系统的运行逻辑。
这样产品既不会显得过重，也不会牺牲系统的表达力。
•	最终感受:
用户不只是“得到一个答案”，而是感受到：
系统正在替自己组织研究、拆解问题、调用能力、汇总判断。

## 8. Design Principles
•	极简优先
即使系统内部很复杂，用户界面也应尽量简洁，不让复杂性直接压到用户脸上。
•	展示复杂，而不是制造复杂
右侧 canvas 的多个终端的目的是帮助用户理解系统，而不是把所有底层实现细节全部暴露出来。
•	结果优先，过程可见
用户首先得到清晰结论；过程是可以查看的，不是强制阅读的。
•	科技感但不赛博噪音
风格上可以偏暗黑、现代、Mac-like、优雅，但不要做成过度拥挤的 hacker UI。
•	适合答辩展示
这个产品不仅要“能用”，还要“能演示”，让评审一眼看出：这不是一个普通聊天机器人，而是一个多 agent research system。

## 9. Success Criteria
•	用户能够自然地把它当作一个 AI research assistant 来使用
•	用户能够明显感知到系统内部存在多 agent 协作，而非单一模型一次性生成
•	用户能够理解 ontology / skills / decision trace 在产品中的角色，而不觉得它们只是技术噱头
•	在右侧 canvas 展开的情况下，系统运行过程具有足够的观赏性与解释性，适合答辩展示
•	在右侧 canvas 收起的情况下，产品依然保持简洁、流畅、可直接使用
•	整体体验能够体现“结构化研究对象 + 多 agent 协作 + 可观察决策过程”这一核心差异点

## 10. v0 Boundaries
	•	v0 聚焦单轮或短周期的股票研究问答
	•	v0 默认使用固定数量的核心 agent，而不是开放式 agent marketplace
	•	v0 中 decision trace 以“核心路径展示”为主，不追求暴露全部底层细节
	•	v0 强调产品形态与系统逻辑的统一，不追求覆盖所有金融研究场景
	•	v0 以“好看、好讲清楚、能跑通”为优先，而不是做成专业交易系统
